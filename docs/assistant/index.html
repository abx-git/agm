Blueprint Pattern · Guided Notebook

Architektur-Doku in dein Repository — Schritt für Schritt

Diese Seite führt dich wie ein Jupyter Notebook: erst verstehen, dann einrichten, dann nutzen. Alles läuft lokal in deinem Git-Repo und deiner AI-IDE — kein Cloud-Dienst.

Md

Was ist das Blueprint Pattern?

Das Problem: AI-Agenten sind gut in einzelnen Dateien, scheitern aber an Querschnittsthemen — wer ruft wen auf, welche ADR gilt, was ändert sich am Interface?

Die Lösung: Du legst in deinem App-Repository einen verlinkten Markdown-Graphen unter docs/architecture/ an. Der Agent ist Schreiber, nicht autonomer Architekt: er pflegt Docs anhand klarer Workflows, du prüfst und entscheidest.

In einem Satz: Einmal Core-Regeln installieren, pro Chat einen Workflow wählen, der Agent schreibt in docs/architecture/ — Fortschritt steht in blueprint.md.

Was du am Ende hast: versionierte Architektur-Doku (arc42 o. Ä.), die mit dem Code mitläuft — gepflegt per git diff und gezielten Agent-Sessions, ohne externes Wiki oder RAG.

Md

Einmalig einrichten (~30 Minuten)

Alles passiert in deinem Anwendungs-Repository (nicht nur im Pattern-Repo). Du brauchst:

  • Ein Git-Repo mit deinem Code
  • Eine AI-IDE (Cursor, Claude Code, Copilot, …)
  • Das Pattern-Repo als Vorlage — klonen oder Submodule
  1. Pattern-Repo bereitstellen
  2. Doc-Templates ins App-Repo kopieren
  3. always-on.md ausfüllen (App-Fakten)
  4. Core-Prompt in IDE-Regeln
  5. Workflow-Skript bereitstellen
In[1]
Schritt 1 — Pattern-Repo holen 
# Neben deinem App-Repo klonen (oder als Submodule)
git clone https://github.com/abx-git/blueprint-pattern.git

# Optional: Submodule im App-Repo
# git submodule add https://github.com/abx-git/blueprint-pattern.git blueprint-pattern

Ab jetzt nennen wir den Pfad zum Pattern-Repo blueprint-pattern/.

In[2]
Schritt 2 — Doc-Struktur ins App-Repo kopieren 
# Im Root deines App-Repos (Pfade anpassen)
mkdir -p docs/architecture

cp -R blueprint-pattern/docs/templates/architecture/* docs/architecture/

# Workflow-Skript + Prompts (für checkout-Befehl)
mkdir -p scripts prompts
cp blueprint-pattern/scripts/bp-workflow.sh scripts/
cp -R blueprint-pattern/prompts/workflows prompts/
chmod +x scripts/bp-workflow.sh
Md

Schritt 3 — always-on.md ausfüllen

Datei: docs/architecture/context/always-on.md

Dort stehen Fakten, die der Agent jede Session kennt — ohne dass du sie in den Chat kopierst:

  • App-Name, Zweck, Tech-Stack
  • Wichtige Pfade (src/, Services, …)
  • Source Map: welche Docs zu welchem Code gehören
In[3]
Schritt 4 — Core-Prompt in die IDE (einmalig) 
# Cursor: Inhalt von blueprint-pattern/prompts/core/system-prompt.md
# in .cursor/rules/ übernehmen (siehe blueprint-pattern/.cursor/rules/)

# Claude Code: in CLAUDE.md verlinken
# Copilot: .github/copilot-instructions.md

# Datei zum Kopieren:
# blueprint-pattern/prompts/core/system-prompt.md

Der Core-Prompt definiert Verhalten (Schreiber, Read-Order, Invarianten) — nicht die Aufgabe des heutigen Chats.

Out[1]

Was du danach in deinem Repository hast

dein-app-repo/
├── docs/architecture/          ← der Architektur-Graph
│   ├── blueprint.md            ← Fortschritt & Session-Log (nicht die Architektur selbst)
│   ├── entry-point.md          ← Einstieg in den Graphen
│   ├── context/
│   │   └── always-on.md        ← App-Fakten (von dir ausgefüllt)
│   ├── prompts/                ← role-*.md (Prozedur, lädt der Agent)
│   ├── work/                   ← Fragen, Analysen, Designs, Reviews
│   ├── interfaces/             ← exports.md, imports.md
│   ├── ops/                    ← Runbooks (optional)
│   └── arc42/                  ← oder c4-light/, adr-first/, …
├── prompts/workflows/          ← Session-Prompts + ACTIVE.md
└── scripts/bp-workflow.sh      ← Workflow umschalten
Checkpoint: Templates liegen im App-Repo, always-on.md ist ausgefüllt, Core-Prompt ist in der IDE. Jetzt startest du die erste Agent-Session.
Md

So funktioniert jede Session

Drei Prompt-Typen — bewusst getrennt, damit der Chat klein und fokussiert bleibt:

TypWoWer setzt esZweck
Core IDE-Regeln Du, einmalig Verhalten: Schreiber, Invarianten
Workflow ACTIVE.md bp-workflow.sh checkout Aufgabe heute: bootstrap, maintenance, …
Role docs/architecture/prompts/role-*.md Agent lädt selbst Schritte & Qualitäts-Gates 
Core (einmal) + Workflow (dieser Chat) → Agent → docs/architecture/ → blueprint.md aktualisieren

Dein Ablauf pro Session:

  1. Im Terminal: ./scripts/bp-workflow.sh checkout <workflow-id>
  2. Neuen Agent-Chat öffnen (bei Review zwingend)
  3. Optional: Session-Prompt aus diesem Notebook kopieren & einfügen
  4. Am Ende: Agent-Ausgabe prüfen ([[ANCHOR:…]]), blueprint.md lesen
▶[4]

Erste Session — Dokumentation anlegen (bootstrap-init)

Der Agent erstellt blueprint.md, wählt ein Template (arc42, …) und schreibt die ersten Kapitel.

1 Checkout im App-Repo

./scripts/bp-workflow.sh checkout bootstrap-init

2 Neuen Agent-Chat öffnen — optional diesen Zusatztext senden:

Lade Session-Prompt…

3 Typische Agent-Antwort am Session-Ende:

[[ANCHOR:TEMPLATE_SELECTED]] arc42
[[ANCHOR:CHANGED_FILES]] blueprint.md, entry-point.md, arc42/introduction.md
[[ANCHOR:PHASE_STATUS]] phase 0 [x], phase 1 [~]
[[ANCHOR:OPEN_QUESTIONS]] Payment: extern oder intern?
[[ANCHOR:LINK_CHECK]] pass

Du prüfst die Docs — Phasen nicht auf [x] setzen, wenn der Inhalt noch dünn ist.

4 Nächste Session

./scripts/bp-workflow.sh checkout bootstrap-continue
