Blueprint Pattern

Architecture documentation in Git — AI-assisted, human-governed

The Blueprint Pattern is not a product or hosted service. It is a repository convention: a linked Markdown graph under docs/architecture/, maintained through scoped agent sessions with explicit workflows. Architecture knowledge stays versioned alongside code; the architect retains decision authority.

1 Understand Scope and boundaries → 2 Adopt ~30 min, one-time → 3 Operate Agent sessions
1

What the pattern provides — and what it does not

In scope

  • Structured architecture docs under docs/architecture/ in your application repository
  • Explicit links between views, interfaces, decisions, and source (Markdown graph)
  • Session-scoped agent tasks (workflows) with traceable outputs
  • Progress and session state in blueprint.md — separate from the architecture content itself

Out of scope

  • No cloud platform, no external wiki, no RAG pipeline
  • Not an autonomous architect — human review and approval remain mandatory
  • Not a substitute for arc42, C4, or ADRs — it hosts and links them in-repo
  • Not a replacement for architecture judgment or sign-off processes
2

One-time adoption — in your application repository

Adoption uses two repositories: this pattern repository (templates and tooling) and your application repository (where all architecture artefacts live). Execute the commands below from your application repository root, adjusting paths as needed.

Prerequisites: Git · AI-enabled IDE (e.g. Cursor, Claude Code) · approximately 30 minutes

A
Clone the pattern repository 
git clone https://github.com/abx-git/blueprint-pattern.git

Result: a blueprint-pattern/ directory adjacent to your application project (or add as a submodule).

B
Copy documentation structure and workflow tooling into the application repo 
# Run from your application repository root (adjust path to blueprint-pattern)

mkdir -p docs/architecture
cp -R ../blueprint-pattern/docs/templates/architecture/* docs/architecture/

mkdir -p scripts prompts
cp ../blueprint-pattern/scripts/bp-workflow.sh scripts/
cp -R ../blueprint-pattern/prompts/workflows prompts/
chmod +x scripts/bp-workflow.sh
C

Define application context

Edit:

docs/architecture/context/always-on.md

  • System purpose, boundaries, and technology stack
  • Key source paths and service map
  • Documentation-to-code traceability (source map)

The agent reads this file each session via IDE rules — it does not need to be pasted into every chat.

D
Install core agent rules in the IDE (one-time) 
blueprint-pattern/prompts/core/system-prompt.md

Cursor: adopt into .cursor/rules/ (see blueprint-pattern/.cursor/rules/ for reference).
Other tools: link from CLAUDE.md, Copilot instructions, or equivalent.

The core prompt defines agent behaviour (scribe, invariants, read order) — not the task for a specific session.

Expected repository layout after adoption

your-application/
├── src/ …                          ← existing codebase
├── docs/architecture/              ← architecture graph (new)
│   ├── blueprint.md                ← session backlog & phase state (not the architecture)
│   ├── entry-point.md              ← graph entry point
│   ├── context/always-on.md        ← stable application context
│   ├── arc42/ …                    ← views (populated during bootstrap)
│   ├── interfaces/ …               ← exports / imports
│   └── work/ …                     ← queries, analyses, designs, reviews
├── prompts/workflows/              ← session prompts; ACTIVE.md set by checkout
└── scripts/bp-workflow.sh          ← activates the current workflow

4

First session — establish architecture documentation (bootstrap-init)

The agent creates blueprint.md, selects a documentation template (arc42 by default), and drafts initial sections. Two actions on your side:

1
Terminal — activate workflow bootstrap-init
./scripts/bp-workflow.sh checkout bootstrap-init
2
IDE — open a new agent chat; optionally paste the session prompt:
Loading…
3
Review — validate outputs and resolve open assumptions 
Typical end-of-session report:

"I created blueprint.md and entry-point.md.
 Template: arc42. Assumption: payment is an external service — please confirm."

→ Confirm or correct in chat. Review new files under docs/architecture/
  before marking bootstrap phases complete in blueprint.md.

Subsequent bootstrap work: ./scripts/bp-workflow.sh checkout bootstrap-continue — the agent resumes the next open phase from blueprint.md.
